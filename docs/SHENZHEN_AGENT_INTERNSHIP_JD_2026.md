## 高频能力结论

按上述样本做人工归纳，岗位能力可分为四层：

1. **核心必需**：Python、LLM API、Prompt、RAG、Tool/Function Calling、Agent workflow、FastAPI/后端接口、基本算法与数据结构。
2. **明显区分度**：asyncio、多 Agent、状态机/工作流、结构化输出、自动评测、重试/超时/限流/日志、Docker/Linux、真实外部 API 和数据库。
3. **后端型岗位加分**：Redis、Postgres/MySQL、Celery/消息队列、Go、分布式缓存和任务系统。
4. **算法型岗位加分**：PyTorch、SFT/LoRA/RLHF/GRPO、多模态、顶会论文。它不是“AI 应用开发”所有岗位的统一硬门槛。

所以你的判断只有一半需要修正：Go 和 Redis 确实会扩大后端型 Agent 岗的覆盖面；SQLite 并不欠缺——第二项目已经把它用于 runs/tasks/evidence/claims/traces/repairs 的规范化持久化，现在又增加了 web search TTL cache。下一步应当把 SQLite 的事务、索引、WAL、并发边界讲清楚，而不是再做一个浅层 CRUD。

## 两个项目必须这样区分

| 维度 | 项目一：高精铝箔工业 Agent | 项目二：DeepResearch Agent |
|---|---|---|
| 核心问题 | 垂直工业现场的受控诊断与操作闭环 | 开放域复杂问题的检索、证据与可信生成 |
| 数据源 | SOP/知识库、设备事件、MES/SPC、mock/sandbox/SQLite/external data contract | 本地 Markdown/PDF corpus、Tavily/Wikipedia/SearXNG、SQLite search cache |
| 编排 | LangGraph 受控路由与工具执行 | 自研 Planner + DAG + TaskStateMachine + Coordinator |
| 关键工具 | RAG、设备数据、SPC、控制图、报告、工单、SOP | Searcher、Reader、Writer、Verifier、Red-Blue、评测器 |
| 可信边界 | 数据 lineage、审批/复核、安全措辞、工业建议不替代工程师 | claim → citation → chunk → quote → verification trace → repair action |
| 最强简历卖点 | ToB 垂域 Agent、结构化工具调用、工业数据接入彩排 | Agent 编排、真实多源检索、自动评测、可追踪与幻觉修复 |
| 不应重复强调 | 不要再把“多 Agent/通用搜索”作为主卖点 | 不要再把“工业数据/SPC/工单”带进来 |

一句话区分：项目一证明“我能把 Agent 接进一个受约束的真实业务流程”，项目二证明“我能从零实现一个可搜索、可验证、可评测的 Agent runtime”。

## 补强顺序

### P0：先把第二项目的真实数据链路讲实

- 已完成：Tavily 原始正文、Wikipedia、arXiv、GitHub README、SearXNG、组合 provider、查询清洗和 source diversity。
- 已完成：安全 HTML/PDF fetcher、SSRF/重定向防护、fetch status、content hash、retrieved_at、失败原因和 Prompt Injection 风险标记。
- 已完成：本地 PDF 逐页提取、page-aware stable chunk id 与 `p. N` citation locator；真实 6 页 ASCC 论文生成 64 chunks 并完成页码级检索验证。
- 已完成：Web 流式上传、10 MiB 限制、格式/magic/UTF-8 校验、SHA-256 去重、原子 corpus 发布和上传 lineage；真实 ASCC PDF 经上传 API 后直接完成研究 run。
- 已完成 12 案例 live-source eval，并增加每周/手动 GitHub Actions monitor、跨 run history cache、趋势 Markdown 和失败 artifact；当前只有 1 次真实 snapshot，需持续积累后才可讨论 SLA。
- 已完成 Searcher 独立 Recall@K/MRR/nDCG/All-relevant@K 回归与 lexical/vector/hybrid 消融；hashing hybrid Recall@5 0.843，并明确 9 个漏召案例和非盲测边界。
- 已接入OpenAI-compatible batch embedding + WAL SQLite content/model cache，并用百炼`text-embedding-v4`完成35题实跑：vector Recall@5 0.943、MRR@5 0.910，相对hashing vector在15题改善且0题退化；该结果是项目内回归集，不包装成外部盲测或线上准确率。

### P1：用 Redis 补“服务化”，不要改写算法

- 已抽象 `SearchCache`，实现 `SQLiteSearchCache`、`RedisSearchCache` 和 Redis → SQLite 故障回退。
- Redis 只承担 TTL cache，不作为 canonical evidence store；SQLite 仍是可审计事实库。
- 已增加 Redis Docker Compose、cache hit/backend/error/fallback lineage；SQLite cache 启用 WAL 与 busy timeout。
- 已将 Web Demo 的进程内 RUNS 字典替换为 WAL SQLite schema v4 durable run store；除兼容 BackgroundTasks 外，已支持独立 worker 的 FIFO 原子领取、ownership heartbeat、lease expiry 重排和最大尝试失败。本地进程级 E2E 已跑通，但仍是单机共享 SQLite/volume，不是跨主机分布式队列。
- 已为任务创建增加 `Idempotency-Key` + request fingerprint，正确区分 replay 与 payload conflict；使用 `BEGIN IMMEDIATE` 原子完成容量检查和插入，超限返回 429/Retry-After，避免无限后台任务。
- 已增加依赖无关的 Prometheus `/metrics`，使用 route template 控制 label cardinality，输出 HTTP count/latency、SQLite run status、active capacity 和 uptime；明确进程指标与共享状态指标的多 worker 边界。

### P1：补部署与稳定性

- 已完成 Provider 并发上限、GitHub 请求间隔、超时、指数退避、熔断与结构化遥测；已补非 root Dockerfile、app/worker/Redis Compose、liveness/readiness、统一 request id 和 JSON 访问/任务日志。当前机器缺 Docker CLI，容器真实 build 是待办验证项。
- Web Demo 展示搜索 provider、cache 命中、外部证据比例和失败降级。

### P2：Go 做小型网关，不重写主项目

- 建议做一个独立的 `search-gateway`：Go + HTTP，统一 Tavily/SearXNG 调用、并发扇出、超时、限流和 Redis cache；Python Agent 只调用统一接口。
- 控制在可独立讲清楚的体量，并写压测对比。这样既补 Go，又不会让第二项目主线从“可信 DeepResearch”变成语言练习。
